# MXMoE-Adapt 技术方案

## 1. 设计原则

1. C500 真机数据是唯一性能事实来源。
2. 原始 FlagGems 基线永久保留，不覆盖、不重命名为优化结果。
3. 运行时只消费已验证配置；示例和预测值不能进入自动派发。
4. 环境、代码、路由、Shape 和结果必须可追溯。
5. 首期不把多卡通信和量化混入核心链路。

## 2. 特征空间

工作负载特征由静态Shape和动态路由组成：

```text
x = [M, E, H, I, top_k, dtype,
     empty_ratio, max/mean load, CV, normalized_entropy, padding_ratio]
```

其中 `M` 为 token 数，`E` 为专家数，`H/I` 为隐藏层与中间层维度。动态特征可区分相同静态Shape下的均衡与偏斜请求。

## 3. 联合搜索空间

搜索变量：

```text
align_block_size
BLOCK_SIZE_M / BLOCK_SIZE_N / BLOCK_SIZE_K
GROUP_SIZE_M
num_warps / num_stages
```

第一阶段用共享存储代理模型过滤明显非法配置；该模型不是编译器真值。第二阶段在C500上进行编译验证。第三阶段使用真实路由实测，按联合目标排序：

```text
J = measured_latency
  + lambda_padding * padding_ratio
  + lambda_compile * compile_time
```

生产配置最终以实测延迟为主，填充率用于解释和跨路由排序，编译成本只作为离线调优的轻量惩罚。

### 多保真搜索

1. 静态资源代理排除明显超预算的Tile。
2. 小Shape只验证编译可行性和数值正确性。
3. 合成均衡/偏斜路由做低成本筛选。
4. 对候选Top-N执行真实Top-K路由回放。
5. 仅对最终候选进行模型层和端到端验证。

真实路由回放固定expert IDs和router weights，使baseline与candidate面对同一动态负载，形成更公平的反事实比较。

## 4. 配置生命周期

配置状态：

```text
candidate -> compiled -> correctness_passed -> measured -> verified
                                                |
                                                v
                                      stale / rejected
```

只有 `verified` 状态可进入运行时数据库。以下事件会使配置变为 `stale`：

- Driver、MXMACA、mcPytorch、mcTriton/FlagTree、FlagGems提交变化；
- 守护Shape的延迟回退超过默认5%；
- 正确性阈值或上游接口发生变化。

## 5. 派发方法

派发先过滤 dtype 和 `verified` 状态，再优先匹配环境指纹，最后计算对数Shape距离并加入路由类型惩罚。如果最小距离超过阈值，返回空配置，由上层回退到 FlagGems 通用实现。

这是一种保守的置信派发。后续积累足够C500样本后，可将距离函数替换为可解释的决策树或代价模型，但首期不使用难以验证的黑盒模型。

## 6. Kernel发展路线

- M0：复用FlagGems通用Kernel，完成C500基线和联合配置调优。
- M1：为小token/高专家数场景开发减少无效块的Decode路径。
- M2：为中大token场景开发提升矩阵利用率的Prefill路径。
- M3：评估对齐、激活和GEMM之间可安全实现的融合边界。
- M4：在首期稳定后评估FP8/W8A8和多卡Expert Parallel。

## 7. 正确性策略

- PyTorch Reference使用清晰逐token逐expert实现。
- 固定随机种子，保存Top-K IDs和权重。
- 分别报告最大绝对误差、最大相对误差和必要时的均方误差。
- FP16/BF16阈值由C500实际数值行为和FlagGems上游测试共同确定。
- 任何不通过正确性的配置不得进行性能排名。

## 8. 性能策略

- 先预热再同步设备计时；记录warmup和repeats。
- 小Shape增加重复次数，报告中位数、P90和P99。
- 同一实验进程交替运行baseline与candidate，降低温度和系统漂移影响。
- 至少重复三轮并保存原始JSON。
- 同时报告延迟、TFLOPS（可合理计算时）、padding ratio和显存峰值。
